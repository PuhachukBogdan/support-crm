# Security Review — Chatwoot as the Base for Our Internal CRM

**Purpose:** Determine whether the Chatwoot open-source codebase is a safe technical foundation for our internal CRM, before we extend it with custom business features and deploy it to our infrastructure.
**Method:** Read-only static review of the full repository (backend, frontend, dependencies, Docker/deployment configuration). No code was modified. All findings below were verified directly against the source — file and line references are included so any item can be independently confirmed.

---

## 1. Security Review Summary

**Overall risk level: Medium-High.**

This rating is driven by a small number of specific, high-severity bugs — not by a flawed architecture. The core trust model (multi-tenant account isolation, role-based access control, SSRF protection on outbound requests, MFA/JWT design, SQL parameterization) is well-built and does not need to be reworked.

**Is the project safe to use as a base for our internal CRM?**
**Yes, conditionally.** It is safe to begin architecture-level customization now. It is **not** safe to deploy with real company or customer data until the blockers below are closed.

**Main risks and blockers:**
1. **An unauthenticated file-upload endpoint** lets anyone on the internet request a signed upload URL for any account/conversation, with no login required.
2. **Super Admin login silently skips two-factor authentication**, even for accounts that have 2FA enabled — a password alone is sufficient for the highest-privilege role in the system, which also unlocks a background-job admin console.
3. **A confirmed cross-site scripting (XSS) vulnerability** in the AI assistant's "mentions" rendering, reachable via AI-generated text derived from customer messages.
4. **Several high-severity configuration and dependency issues**: password-reset links that never expire, two channel webhooks (SMS, Twilio Voice) that accept unsigned/forged requests, a production database password that silently falls back to a well-known default value if not explicitly set, HTTPS not enforced by default, an end-of-life framework version with no further security patches, and ~65 known vulnerabilities in frontend JavaScript dependencies (including a widely-used HTTP library that is a direct dependency).
5. **No data-retention policy and no built-in database backup mechanism** — conversations, messages, and attachments are kept indefinitely by default, and there is no shipped tooling for backing up the database at all. Both are compliance gaps for a CRM holding company/customer data.

**What is already solid** (verified, not assumed): tenant data isolation, outbound-request (SSRF) protection, zero known vulnerabilities in Ruby-side dependencies, sound MFA/JWT implementation, reasonable CSRF/CORS defaults, no debug endpoints or API documentation exposed in production, and real-time (WebSocket) events correctly withhold private internal notes from customer-facing connections.

**A non-technical blocker that requires a business decision, not a code fix:** part of the codebase (audit logging, single sign-on, custom roles, billing) sits under a separate "Enterprise" license that requires a **paid Chatwoot subscription for any production use — including purely internal deployments**, not just resale. If these features are needed, budget for the subscription or plan to build equivalent functionality under our own code before going live.

## 4. Hardening Recommendations

### Required before internal deployment (blockers)
1. Fix the unauthenticated file-upload endpoint (#1).
2. Fix the Super Admin two-factor-authentication bypass (#2) — this also reduces the risk of the background-job admin console exposure (#13).
3. Fix the confirmed XSS in the AI assistant's mention rendering (#3).
4. Fix the password-reset token expiration check (#4).
5. Add request-signature verification to the SMS and Twilio Voice webhooks, or disable those channels if they are not in use (#5, #6).
6. Remove the silent default database password; require an explicit, non-trivial password everywhere, including in deployment templates (#11).
7. Enforce HTTPS by default (#12).
8. Upgrade off the end-of-life framework version (#10).
9. Update the flagged high/critical-severity frontend dependencies (#14).
10. **Decide the Enterprise-license question** (#47) — this decision also determines how the audit-trail gap (#28) gets resolved.

### Recommended before/during customization
- Define a data-retention policy and implement a documented database backup procedure — neither exists today (#7, #16).
- Close the remaining file-upload validation gap and standardize attachment validation across all channels (#15).
- Add expiration/rotation to API access tokens and enable account lockout after repeated failed logins (#18, #19).
- Extend personal-data filtering in logs and stop logging full third-party response bodies on failure — required before enabling any centralized log aggregation (#20).
- Close the contact-export and report/survey-export gaps: link expiration, audit logging, rate limiting (#21, #22).
- Enable a baseline Content-Security-Policy (#24).
- Run application containers as a non-root user and pin production deployments to a fixed release version (#25).
- Validate the origin of cross-frame messages in the widget and dashboard-app integrations (#26).
- Scope browser-stored drafts and search history to the specific account/user, and clear them on session expiry — important if the system will run on shared support-desk computers (#29, #32).
- Fix the private-note "typing" indicator leak (#30).

### Optional / later phases
- Raise the minimum password length; use constant-time comparison for the one HMAC check identified; add regression tests around the SQL filter-builder's safe-value list; pin the two branch-sourced dependencies to fixed versions; strip debug logging from production frontend builds; align the diagnostic-preview-page exclusion to cover staging as well as production; pin the cache-service image to a specific version; add database-connection encryption if the database is ever split onto a separate host.
- Build a formal data-export/portability feature only if a specific future compliance requirement calls for it.
- Obtain a brief legal sign-off on the two LGPL-licensed dependencies given future resale/white-labeling plans — a formality, not a blocker.

## 5. Go / No-Go Recommendation

**Can we proceed with customization?**
**Yes, now.** The underlying architecture — multi-tenancy, role-based access control, outbound-request protection, authentication design — is sound and does not require a rewrite. Nothing found in this review is a reason to evaluate a different base product.

**Can we deploy it internally after fixes?**
**Yes**, once the items listed under "Required before internal deployment" are closed. Most are small, targeted code or configuration changes — realistically hours to a few days of focused work, not weeks. The two Critical-severity items are each a small, well-understood fix once identified.

**Which issues must be fixed before production use with real company or customer data?**
- All Critical- and High-severity findings (#1–#14 in the table above).
- The Enterprise-license decision (#47) — not a technical bug, but a real compliance blocker if left undecided while relying on Enterprise-only features in a live deployment.
- Do **not** enable bulk contact/report export, centralized log aggregation, or the AI assistant feature against real customer data until their related findings (#3, #20, #21, #22, and the AI-provider row in the External Communications Map) are addressed.

**Bottom line: GO**, conditional on closing the items above before production go-live. This is a well-architected open-source product that needs a focused, well-scoped hardening pass — not a reason to look for a different foundation.

## Reviewer's Recommendation

Going back to the original task: the goal was to confirm whether this codebase is *safe to use as the foundation for an internal CRM that may process sensitive company and customer data*, and to flag anything that could create a security or compliance problem after deployment.

**My recommendation: proceed with Chatwoot as the base.**

The reasoning:

- **Nothing found here is a reason to reject the product.** Every Critical/High finding is a specific, isolated, well-understood bug or misconfiguration — not evidence of a broken security model. The parts that are hardest to retrofit after the fact (multi-tenant data isolation, role-based access control, the authentication/token architecture, outbound-request protection) were all reviewed and hold up. That is the part that would have made me recommend walking away, and it didn't happen.
- **The blockers are cheap relative to the alternative.** Fixing the two Critical bugs and the High-severity list realistically takes a small, bounded engineering effort (days, not weeks) before any real data touches the system. Rebuilding an equivalent CRM from scratch, or vetting a different open-source base to the same depth, costs materially more and carries its own unknown risks that haven't been surfaced by a review yet.
- **The one item that isn't a simple code fix — the Enterprise-license question (#47) — needs a decision now, not later.** It doesn't block starting customization, but it should be decided before scope is planned, because it changes how much custom engineering work (audit logging, SSO, roles) the team will actually own.
- **Caveat:** this recommendation assumes the "Required before internal deployment" list in Section 4 is actually completed before real company or customer data enters the system, and that AI-assistant/bulk-export features stay off for real data until their specific findings are closed.
